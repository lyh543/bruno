## 🤖 AI 编码与测试准则 (Project Guidelines for LLM)

### 1. 行为驱动测试原则 (Behavior-Driven Testing Standards)

**核心指令：** 必须严格遵守 `CODING_STANDARDS.md` 中的测试规范。在编写、重写或评审测试时，确保测试关注的是**业务行为和接口契约**，而不是内部实现细节。测试必须是可维护的、健壮的，且能够安全地**并行执行**。

* **❌ 严禁编写的坏测试（反面教材）：**
* **测试实现细节：** 比如测试某个私有方法是否被调用，或者断言内部状态变量的精确变化。这种测试会导致重构代码时（行为没变，实现变了）测试大量崩溃。
* **共享可变状态：** 严禁在测试间共享全局变量、数据库连接、单例状态或静态上下文中未隔离的数据。
* **硬编码环境依赖：** 使用固定的本地文件路径、硬编码的端口号，或者依赖特定的本地时区/时间。


* **✅ 必须遵循的好测试（正面规范）：**
* **面向行为 (Behavioral)：** 遵循 `Given-When-Then` 结构。测试应当描述：“在 [特定上下文] 下，当 [调用公开行为] 时，应该 [产生什么可观测的结果/输出]”。
* **无副作用 (Side-effect Free)：** 每个测试运行前后，必须通过 `Setup` 和 `Teardown` 彻底清理自身产生的副作用。
* **完全隔离与并行化 (Parallel-Safe)：**
* 使用随机或动态生成的唯一标识（如 UUID、随机端口、动态内存数据库名）来隔离数据。
* 依赖外部时间时，必须通过依赖注入（如 `IClock` 或 Mock 时间）来控制，严禁直接调用系统当前时间。





> 💡 **LLM 思考要求：** 在你修改任何核心逻辑后，必须主动审查相关测试。如果发现现有测试违反了上述并行或行为原则，请**先批评并重构测试**，再提交逻辑修改。

### 1.1 E2E / Playwright 约定 (E2E / Playwright Guidance)

* **核心指令：** 对 Electron/Workspace 相关流程补 E2E 时，优先复用现有测试驱动方式与最邻近 spec，避免新造 fixture 或引入额外的全局测试钩子。
* **优先策略：**
1. 优先把新用例补到相邻行为的现有 spec 文件中。
2. 对系统目录选择、文件选择等原生弹窗，优先在 Electron 进程里 stub `dialog.showOpenDialog` 等 API，而不是尝试驱动原生窗口。
3. 断言默认名称时，优先基于真实临时目录 basename 等可观察输入推导，不要硬编码测试运行时会变化的目录名。
4. 单条本地调试时，优先使用不会启动 HTML report 服务的运行方式，例如：`PLAYWRIGHT_HTML_OPEN=never npx playwright test ... --reporter=list`。

* **禁止事项：**
* 不要为了覆盖单个 workspace 行为去新增复杂的 test harness 或长期驻留的全局 stub。
* 不要依赖 Playwright 失败后自动打开/托管 HTML report 作为调试主路径；先保证命令行输出可直接定位失败。

### 2. UI 变更与组件复用规范 (UI & Component Reuse Standards)

**核心指令：** 修改或新增 UI 视图时，**复用是第一优先级**。严禁盲目手写原生样式（如自定义 CSS/Inline Styles）或凭空创造新的低级原子组件。你必须像一个经验丰富的资深前端/客户端工程师一样，优先在现有组件库中寻找拼图。

* **❌ 拒绝野蛮生长：**
* 严禁在未检索现有组件的情况下，直接使用原生标签（如 `<div>`, `<span>` 或平台原生基础控件）去手写布局、边距、颜色和阴影。
* 严禁为了一个微小的视觉差异，去复制粘贴一个现有的组件并魔改它，造成代码重复。


* **✅ 严格复用链路：**
1. **全局设计系统 (Design System)：** 优先使用项目中已引入的通用组件库（如项目特定的 UI 库/第三方标准库）。
2. **局部业务组件：** 在修改特定页面时，先检索当前模块及邻近目录下是否已有现成的局部组件、`DataTemplate` 或布局样式。
3. **属性配置优先：** 如果现有组件无法完全满足需求，优先通过阅读现有组件的公开属性（Props/Properties）或样式扩展点来达成目标，而不是重写。
4. **组件目录索引：** 在 `packages/bruno-app/src/components/readme.md` 中先查组件目录与复用建议，再决定是否新增组件。


* **🎨 必须手写样式时的唯一例外：**
* 只有当现有的所有组件和设计系统**绝对无法满足**全新的视觉/交互需求时，才允许编写新样式。
* **要求：** 编写前必须显式向用户说明：“我检索了现有组件库，未发现满足 X 需求的组件，原因在于 Y，因此我准备新建一个组件/样式...”。

### 🔍 LLM 提交前自我检查清单 (Pre-flight Checklist)

在你将代码交付给用户之前，请在心底（或思考链中）默念并确认以下问题：

1. **我刚刚写的测试，如果 100 个实例同时并发运行，会因为抢占某个资源而报错吗？**
2. **我修改的这个测试，在 3 年后别人重构内部函数名、但业务逻辑没变时，会莫名其妙挂掉吗？**
3. **我刚刚加的这个 UI 间距/颜色，能在现有的 Theme/Style/Component 中找到对应的变量或组件吗？我是不是又偷偷写了硬编码的样式值？**
